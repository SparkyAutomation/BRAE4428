Dataset Fundamentals

Training, Validation & Test Data

Every ML project splits its dataset into three non-overlapping subsets. Each plays a distinct role - mixing them up is one of the most common sources of misleading results.

🟢 Training Set

The model sees this data repeatedly across epochs. Weights are updated via backpropagation. Never evaluate final performance on training data - the model has memorized it.

🔵 Validation Set

Used after each epoch to check how well the model generalizes. You use val loss to decide when to stop training, which learning rate to use, and which architecture works best.

🟡 Test Set

Locked away until the very end. You report test metrics once - after all hyperparameter tuning is done. Treat it like a sealed exam.

How to Read the Curves

Both curves high: underfitting. Train low, val diverging: overfitting - add dropout, augmentation, reduce model size. Both converging low: sweet spot found.

Early Stopping in Practice

Monitor val loss after every epoch. If it has not improved for patience epochs, stop and restore best weights. Patience of 5-15 is typical.

Training Settings - Data Augmentation

Augmentation - Synthetic Variety to Fight Overfitting

Augmentation artificially expands your training set by applying random transformations to existing images. The model sees a different version each epoch, making it harder to memorize and forcing it to learn robust features.

📐 Augmentation Effect on Effective Dataset Size
Without augModel sees same N images every epoch - memorizes after ~20 epochs
With augEach epoch: N images x random transforms - effectively infinite variety
Rule of thumbStrong aug - train longer (more epochs needed to converge)

When to Use Strong Augmentation

Small datasets (<5k images) benefit most from aggressive augmentation. YOLO-style training uses mosaic, random scale, HSV shifts, and flips by default. Too much augmentation on an already-large dataset can slow convergence without benefit.

Augmentation vs Epochs

More augmentation = more variety per epoch = model needs more epochs to converge. A rule of thumb: if you double augmentation strength, add 20-30% more epochs. Watch val loss - if it is still decreasing at your epoch limit, train longer.

Reading the Matrix

Each row = actual class. Each column = predicted class. A perfect model has all values on the diagonal and zeros everywhere else. Off-diagonal values reveal which classes get confused with each other.

Normalized vs Raw Counts

Raw counts show absolute errors. Normalized (row-wise) shows the rate of confusion per class - useful when classes are imbalanced.

Precision vs Recall Tradeoff

High precision, low recall: conservative - only fires when very confident. Good for spam filters. High recall, low precision: catches almost everything but has many false alarms. Good for cancer screening. F1 punishes extreme imbalance between the two.

When Accuracy Lies

Imagine 99% of emails are not spam. A model that always predicts "not spam" gets 99% accuracy - but 0% recall on spam. This is the accuracy paradox. On imbalanced datasets, always report precision, recall, and F1 per class.

Training Configuration

Hyperparameters - The Knobs You Turn

Parameters are the weights the model learns during training. Hyperparameters are the settings you choose before training starts - they control how learning happens, not what is learned.

📐 Parameters vs Hyperparameters
ParametersWeights & biases - learned automatically via gradient descent during training
Learning rateHow big a step to take each gradient update - too high = diverge, too low = slow
Batch sizeHow many samples per gradient update - affects noise and memory usage
EpochsHow many full passes through the training data
Dropout rateFraction of neurons randomly zeroed each forward pass - regularization
Weight decayL2 penalty on large weights - keeps model simple, reduces overfitting

Learning Rate - How Fast Does the Model Learn?

The learning rate controls the step size during gradient descent. Too high and the loss bounces around or explodes. Too low and training crawls. The "just right" rate converges smoothly.

Batch Size - Noisy vs Smooth Gradients

Small batches give noisy gradient estimates - the loss curve is jagged but can escape local minima. Large batches give smooth, accurate gradients but may overfit and use more memory.

💡 Smaller batch = noisier curve - Larger batch = smoother but may overfit

Moderate noise - good default

Dropout - Randomly Silencing Neurons

During training, dropout randomly sets a fraction of neuron outputs to zero. This prevents neurons from co-adapting and forces the network to learn redundant representations.

💡 Adjust dropout rate - grey neurons are dropped this forward pass

0 = no dropout - 0.5 = half dropped - 0.9 = too aggressive

Regularization - L1, L2 & Dropout

L1 Regularization

Adds the sum of absolute values of weights to the loss. Pushes many weights to exactly zero - produces sparse models. Useful for feature selection.

Loss += lambda * sum|w|

L2 Regularization (Weight Decay)

Adds the sum of squared weights to the loss. Penalizes large weights, keeping them small and distributed. The most common regularizer in deep learning - often called "weight decay" in AdamW.

Loss += lambda * sum(w^2)

Dropout

Randomly zeros neuron outputs during training. At inference, all neurons are active but outputs are scaled. Acts like training an ensemble of many sub-networks. Typical rates: 0.2-0.5 for hidden layers.

p(keep) = 1 - dropout_rate

Object Detection Metrics

mAP - Mean Average Precision

mAP is the standard metric for object detection. It combines precision and recall across all confidence thresholds into a single number. mAP@0.5 uses IoU>=0.5 as the match criterion. mAP@0.5:0.95 averages over IoU thresholds from 0.5 to 0.95 in steps of 0.05 - much stricter.

🗣 Plain English - What mAP Actually Means
1
Precision: "When I say I found it, I'm usually right."
2
Average Precision: "For one class, I check precision as recall changes, then average it - that's the area under the PR curve."
3
Mean Average Precision: "Then I average that score over all classes - so every class counts equally."
📐 mAP Calculation Pipeline - mAP = many classes, average their precision curves
Step 1For each class: rank all detections by confidence (highest first)
Step 2For each detection: compute cumulative Precision and Recall
Step 3AP = area under the Precision-Recall curve (interpolated)
Step 4mAP = mean of AP across all classes
mAP@0.5AP computed with IoU threshold = 0.50
mAP@0.5:0.95Average of AP at IoU = 0.50, 0.55, 0.60 ... 0.95 (10 thresholds)
📊 mAP@0.5 = lenient (box just needs to overlap 50%)  ·  mAP@0.5:0.95 = strict (rewards tight, well-placed boxes)

mAP@0.5 vs mAP@0.5:0.95

mAP@0.5 is the classic PASCAL VOC metric - a detection counts as correct if IoU >= 0.5. mAP@0.5:0.95 (COCO metric) averages over 10 IoU thresholds, rewarding models that localize tightly. A model can score 0.85 mAP@0.5 but only 0.55 mAP@0.5:0.95 if its boxes are sloppy.

What's a Good mAP?

Context matters. YOLOv8n on COCO: ~37 mAP@0.5:0.95. YOLOv8x: ~53. For a custom dataset with 5 well-defined classes and 2000 images, 0.80+ mAP@0.5 is achievable. Below 0.5 usually means insufficient data, poor labeling, or class imbalance.
